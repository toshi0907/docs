# Copilot Instructions for Documentation Creation

このドキュメントは、GitHub Pagesでの表示に最適化された技術ドキュメント作成のためのガイドラインです。今後の技術ドキュメント作成時は、GitHub Pages環境での最適な表示を考慮して作成してください。

* 目次
{:toc}

## ドキュメント作成の基本方針

### GitHub Pages最適化の重要性
- **Jekyll互換性**: GitHub PagesはJekyllを使用するため、Jekyllと互換性のあるMarkdown記法を使用する
- **自動生成**: GitHub Actionsによる自動ビルドとデプロイを前提とした構造にする
- **レスポンシブデザイン**: モバイル・デスクトップ両方での読みやすさを考慮する

### 1. 構造化とナビゲーション
- **必須TOC配置**: **全てのマークダウンファイルは先頭に目次（TOC）を含める必要があります**
- **TOC形式**: Jekyll/kramdownの自動目次生成機能 `{:toc}` を必ず使用する
- **TOC統一性**: `* 目次` + `{:toc}` の形式ですべてのドキュメントを統一する
- **手動TOC禁止**: 手動で作成したリスト形式の目次は使用せず、必ず `{:toc}` を使用する
- **階層構造**: H1（#）、H2（##）、H3（###）を適切に使用して、情報を論理的に整理する
- **セクション分け**: 関連する内容をまとめ、独立したセクションとして構成する
- **自動更新**: `{:toc}` により見出し構造変更時の目次メンテナンスが不要になる

### 2. 内容の網羅性と段階的学習
- **基本から応用へ**: 基本概念から始めて、徐々に高度な内容に進む構成にする
- **実用的な例**: 理論だけでなく、実際に使用できるコード例や実践的な応用例を豊富に含める
- **参考情報**: デバッグ技法やエラーハンドリングなど、実際の作業で必要となる情報も含める

### 3. コード例とフォーマット

#### コードブロックの使用
```bash
# コメントは日本語で記述
echo "実行可能なコード例を提供"
```

#### 重要なポイント
- **実行可能性**: 提供するコード例は実際に動作するものにする
- **コメント**: コードには適切な日本語コメントを付ける
- **説明文**: コードブロックの前後に説明文を配置する

### 4. 言語とトーン
- **日本語**: 全ての説明は日本語で記述する
- **丁寧語**: 敬語を適度に使用し、読みやすい文体を心がける
- **専門用語**: 必要に応じて英語の専門用語を併記する（例：shebang（シバン））

### 5. 具体的な書式ガイドライン

#### TOCの必須要件と書式
**すべてのマークダウンファイルに必須：**
```markdown
* 目次
{:toc}
```

**重要**: 
- **自動生成TOCの使用**: Jekyll/kramdownの自動目次生成機能 `{:toc}` を必ず使用する
- **手動TOCの禁止**: 手動で作成したリスト形式の目次は使用しない
- **統一性**: すべてのドキュメントで `* 目次` + `{:toc}` の形式で統一する
- **保守性**: `{:toc}` により見出し構造の変更時も自動的に目次が更新される

**非推奨（手動TOC例）:**
```markdown
## 目次
1. [基本概念](#基本概念)
2. [インストール](#インストール)
3. [基本的な使い方](#基本的な使い方)
   - [サブセクション1](#サブセクション1)
   - [サブセクション2](#サブセクション2)
4. [実用的な例](#実用的な例)
```

#### GitHub Pages用の YAML Front Matter（推奨）
```markdown
---
layout: page
title: "ページタイトル"
---
```

#### セクションヘッダー
```markdown
## メインセクション

### サブセクション

#### 詳細項目
```

#### リストの使用
- 箇条書きは`-`を使用
- 番号付きリストは`1.`を使用
- ネストしたリストも適切にインデントする

#### コマンドと実行例
```bash
# 実行権限を付与
chmod +x script.sh

# スクリプトを実行
./script.sh
```

**実行方法:**の形式で実行手順を明確に示す

### 6. 読者への配慮
- **学習段階**: 初心者から上級者まで対応できる内容構成
- **検索性**: 目次とセクション分けにより、必要な情報を素早く見つけられる構造
- **実践性**: すぐに使える実例とサンプルコードを提供

### 7. ドキュメントの保守性
- **更新しやすさ**: セクションが独立しており、部分的な更新が容易
- **拡張性**: 新しい内容を追加しやすい構造
- **一貫性**: 全体を通して統一されたフォーマットと文体
- **GitHub Pages互換**: Jekyll及びkramdownとの互換性を維持

### 8. GitHub Pages特有の考慮事項
- **ビルド時間**: 過度に複雑な構造は避け、効率的なビルドを心がける
- **リンク構造**: 相対パスを適切に使用し、GitHub Pages環境でのリンク切れを防ぐ
- **画像とリソース**: 適切なパスでの画像配置とAlt textの設定
- **SEO対応**: 適切なメタデータとタイトル構造で検索エンジン最適化を図る

## 特記事項

### GitHub Pages向けドキュメント作成の必須要件
1. **TOC必須配置**: すべてのマークダウンファイルに `* 目次` + `{:toc}` を含める
2. **自動TOC統一**: 手動目次は禁止し、Jekyll/kramdownの `{:toc}` 機能で統一
3. **kramdown互換性**: GitHub PagesのMarkdownプロセッサとの互換性を確保
4. **Jekyll最適化**: 適切なYAML Front Matter使用でJekyll処理を最適化
5. **レスポンシブ対応**: モバイル・デスクトップ両対応の構造設計

### 技術ドキュメント作成の具体的なアプローチ
1. **基本概念の確立**: 対象技術の基礎から開始
2. **段階的な複雑化**: 基本→応用→実践の順序で説明
3. **実用性重視**: 実際に使用できるコード例と実践的な応用例を重点的に解説
4. **エラー対応**: デバッグとエラーハンドリングの包含
5. **豊富な例示**: 各概念に対して複数のコード例を提供
6. **GitHub Pages最適化**: Jekyll環境での最適な表示を考慮した構造設計

このガイドラインに従って、GitHub Pages環境で最適に表示される技術ドキュメントを作成してください。

## GitHub Pages最適化チェックリスト

すべての新規・更新ドキュメントについて、以下の項目を確認してください：

### 必須項目
- [ ] **TOC配置済み**: ドキュメント先頭に `* 目次` + `{:toc}` を配置
- [ ] **自動TOC使用**: Jekyll/kramdownの `{:toc}` 機能を使用（手動TOC禁止）
- [ ] **階層構造**: H1/H2/H3の適切な使用
- [ ] **kramdown互換**: GitHub Pagesで正しく表示される形式
- [ ] **config更新**: 新規にファイル追加時は、configファイルのheader_pages:に項目を追加すること
- [ ] **項目順序**: header_pages:の項目の並び順は、index.mdのドキュメント一覧の順序と揃えること

### 推奨項目
- [ ] **YAML Front Matter**: Jekyll用メタデータの追加
- [ ] **レスポンシブ配慮**: 長い行の適切な改行
- [ ] **コードブロック**: 適切な言語指定とコメント
- [ ] **リンク検証**: 相対パスの正確性確認

### 品質確認
- [ ] **可読性**: 段階的で理解しやすい構成
- [ ] **実用性**: 実行可能なコード例の提供
- [ ] **完全性**: 基本から応用まで網羅的な内容
- [ ] **保守性**: 更新・拡張しやすい構造
